I know quite a bit about Android ROMs, kernels, and the broader custom development scene. Here's a comprehensive overview:
Custom ROMs
What they are: Modified versions of Android that replace the stock firmware (the "ROM" — Read-Only Memory, though it's really just firmware).
Types of ROMs
Table
Type	Description	Examples
AOSP-based	Pure Android Open Source Project, minimal Google apps	LineageOS, crDroid, Pixel Experience
Google Pixel-style	AOSP + Pixel-exclusive features	Pixel Experience, PixelOS, Evolution X
MIUI-based	Xiaomi's UI ported to other devices	MIUI ports (less common now)
One UI-based	Samsung's UI ported to other devices	One UI ports (rare, usually device-specific)
GSI (Generic System Images)	Universal ROMs for Project Treble devices	phh-Treble, LineageOS GSI
Popular Custom ROMs
LineageOS — The successor to CyanogenMod. Clean, stable, privacy-focused. Supports 100+ devices officially, many more unofficially.
crDroid — Feature-rich, highly customizable, based on LineageOS with extra tweaks.
Evolution X — Pixel-like experience with tons of customization options.
Pixel Experience — Closest to a real Pixel phone, includes Pixel launcher, camera, and features.
GrapheneOS — Hardened Android focused on security and privacy (Pixel devices only).
CalyxOS — Privacy-focused with microG support (mainly Pixels).
Paranoid Android — Known for unique UI innovations (Halo, Peek, etc.).
ArrowOS — Lightweight, close to AOSP, good performance.
Kernels
What they are: The core of the operating system. It manages hardware resources, CPU scheduling, memory, drivers, and power management. Custom kernels replace the stock kernel to unlock new features or improve performance.
What Custom Kernels Can Change
Table
Feature	What It Does
CPU Governors	Algorithms that control CPU frequency scaling (e.g., schedutil, ondemand, performance, powersave)
I/O Schedulers	Control how disk read/write operations are queued (e.g., noop, deadline, cfq, bfq)
Overclocking/Underclocking	Raising or lowering CPU/GPU frequency limits beyond stock
Undervolting	Reducing voltage to save battery and reduce heat
Custom TCP Congestion Algorithms	Improve network performance (e.g., bbr, westwood)
WireGuard/OpenVPN modules	Built-in VPN support
KSU / APatch support	Kernel-level root solutions
Wake locks control	Better battery management by controlling what keeps the device awake
Sound mods	Audio enhancements, high-res audio support
Popular Custom Kernels
Franco Kernel — Known for balance between performance and battery (historically very popular).
ElementalX — Feature-rich, supports many devices, good for tweaking.
SkyKernel / NeonKernel / Predator Kernel — Device-specific kernels with gaming/performance focus.
KernelSU / APatch compatible kernels — Built for root solutions that don't modify the system partition.
Root Solutions
Table
Tool	How It Works	Notes
Magisk	Systemless root — modifies boot image without touching system partition. Most popular.	Modules system for systemless mods.
KernelSU	Kernel-based root — grants root via kernel, very stealthy.	Requires custom kernel with KSU support.
APatch	Hybrid of Magisk + KernelSU — patches kernel + uses Magisk modules.	Newer, gaining traction.
Recovery
Before flashing anything, you need a custom recovery:
Table
Recovery	Features
TWRP (Team Win Recovery Project)	Most popular. Touch-based GUI, supports backups, flashing zips, ADB sideload.
OrangeFox — TWRP fork with modern UI, built-in Magisk/KSU flashing, better encryption support.
SHRP (Sky Hawk Recovery Project) — Lightweight, good for older/low-end devices.
Lineage Recovery — Minimal, included with LineageOS builds.
Porting ROMs
What it means: Taking a ROM built for one device and adapting it to work on another, usually similar, device.
Why It's Hard
Device trees (device/<vendor>/<codename>/) define hardware-specific code — kernel, blobs, HALs, overlays.
Vendor blobs — Proprietary drivers from Qualcomm, MediaTek, etc. Often device-specific.
Kernel source — Must match the device's hardware. Sometimes shared across similar devices (e.g., same SoC).
Hardware differences — Camera sensors, fingerprint readers, display panels, audio codecs vary even between similar models.
Basic Porting Process (simplified)
Get source — Fork the ROM's device tree for a similar "base" device.
Adapt device tree — Change codename, update kernel defconfig, adjust partitions, fix fstab.
Update vendor blobs — Extract from stock firmware or use prebuilt vendor.
Patch kernel — Add device-specific drivers, fix DTB (Device Tree Blob).
Fix build errors — HAL mismatches, missing dependencies, sepolicy issues.
Test & debug — Boot, check logs (dmesg, logcat), fix what's broken (WiFi, Bluetooth, camera, etc.).
Common Issues When Porting
Camera doesn't work — HAL mismatch, missing proprietary libraries.
No audio — Wrong audio HAL or missing mixer paths.
RIL (mobile data/calls) broken — Modem firmware mismatch.
Encryption fails to mount — Wrong fstab or keymaster HAL issue.
SELinux denials — Need to add sepolicy rules.
Key Concepts & Tools
Table
Term	Meaning
Device Tree (DTB)	Hardware description passed to the kernel at boot.
HAL (Hardware Abstraction Layer)	Interface between Android framework and hardware drivers.
Treble / Project Treble	Android 8+ feature that separates vendor and system partitions, making GSIs possible.
VNDK (Vendor Native Development Kit)	Allows system and vendor to use different library versions.
A/B partitioning	Seamless updates with two system slots.
Dynamic partitions (super partition)	Logical partitions introduced in Android 10.
Fastboot	Bootloader protocol for flashing partitions.
ADB (Android Debug Bridge)	Command-line tool for communicating with Android devices.
Logcat	Android system log viewer.
Dmesg	Kernel ring buffer log.
For Your Samsung A34 5G (SM-A346x)
Samsung devices are notoriously difficult for custom development because:
Bootloader is often locked — Samsung Knox fuse trips permanently on bootloader unlock, voiding warranty and breaking Samsung Pay/Secure Folder forever.
Exynos vs. Snapdragon — Regional variants have different SoCs. The A34 uses MediaTek Dimensity 1080, which has limited custom kernel/ROM support compared to Qualcomm.
No official unlock support in many regions — Samsung stopped providing unlock tokens for some devices.
Odin/Heimdall — Samsung's proprietary flashing tools (instead of fastboot).
Reality check: The Galaxy A34 5G has very limited custom ROM/kernel development. You might find:
Some GSIs via Treble Info app compatibility
Minimal custom kernel work (if bootloader is unlockable in your region)
Mostly stock-based mods (debloated ROMs, One UI mods via Magisk)
If you're interested in ROM development, devices with unlockable bootloaders and active communities (Pixels, OnePlus, Poco, Motorola, older Samsungs with Exynos) are much better starting points.

One UI ports to older devices like the Galaxy S10 are a fascinating but challenging corner of Android development. Here's what you need to know:
What One UI Ports Actually Are
These aren't "ports" in the traditional ROM sense. Because Samsung doesn't open-source One UI, developers can't simply build it from source like LineageOS. Instead, they:
Decompile and modify the stock Samsung firmware (AP file)
Transplant the One UI system image onto an older device's framework
Patch hardware abstraction layers (HALs) to bridge old drivers with new One UI code
Inject older device-specific blobs and kernels
The result is a hybrid franken-ROM — new Samsung UI running on old hardware with significant compromises.
Why the S10 Specifically?
The Galaxy S10 (SM-G973x, exynos9820) is actually a prime candidate for One UI ports because:
Table
Advantage	Why It Helps
Exynos 9820	Samsung's own SoC = better source code availability vs. Snapdragon
Bootloader unlockable	Exynos variants (global) allow BL unlock; US Snapdragon does not
Large user base	More testers, more bug reports, more motivation for devs
Hardware similarity	Shares components with Note 10, S20 FE — easier blob sharing
Treble compliance	Android 9+ Treble makes GSI-style approaches partially viable
How One UI Ports Are Actually Built
Method 1: Full Firmware Transplant (Most Common)
plain
Source: Latest One UI firmware for a similar device (e.g., S21 FE, A54)
Target: S10 (exynos9820)

Steps:
1. Extract system.img, vendor.img, product.img from source firmware
2. Replace kernel with S10-compatible custom kernel (based on Samsung source)
3. Replace device tree and HALs with S10 versions
4. Patch init.rc, fstab, and sepolicy for S10 partition layout
5. Fix RIL (radio interface) for S10 modem firmware
6. Patch camera HAL to work with S10's camera stack
7. Sign and repack as flashable zip or Odin-compatible tar
Method 2: GSI + One UI Overlay
Flash a One UI GSI (Generic System Image) built for arm64_binder64
Pair with S10 vendor image
Less stable, more bugs, but easier to build
Method 3: Port from Similar Samsung Device
Take an official One UI update for a device with similar SoC (e.g., Note 10 Lite with Exynos 9810, or S20 FE with Exynos 990)
Adapt device tree for S10 hardware differences
This is the most successful approach historically
Specific Challenges for S10 One UI Ports
Table
Component	Problem	Typical Fix
Camera	New One UI camera app requires new camera HAL; S10 has old sensors	Port old camera app, or patch libexynoscamera3.so, or use GCam
Fingerprint	Ultrasonic sensor driver changed in One UI 5+	Backport old goodix driver, or disable ultrasonic and use face unlock
Samsung Pay	Knox fuse trips on BL unlock; even if not, SafetyNet fails	Impossible to fix fully; use Google Pay with Magisk Hide/Play Integrity Fix
Always-On Display	New AOD engine expects newer display panel	Disable or use old AOD implementation
DeX	Requires specific HDMI/DisplayPort alt-mode support	Often broken; sometimes works with USB-C hubs
5G	S10 is 4G only; One UI 6+ assumes 5G capability	Patch framework to hide 5G UI elements
Battery optimization	New power management expects newer PMIC	Tweak power_profile.xml, sometimes causes random reboots
Active Projects & Communities
Table
Source	What You'll Find
XDA Developers — Galaxy S10 forums	Most active hub. Look for threads like "One UI 6.1 Port for S10"
Telegram groups	@S10ports, @OneUIPorts, regional dev groups (often more current than XDA)
SamFw / Frija	Tools to download official Samsung firmware for source material
GitHub repos	exynos9820-dev, s10-oneui-port, etc. — device trees and patches
Notable S10 Port Examples (Historical)
One UI 5.1 (Android 13) on S10 — Achieved with decent stability by ~late 2023
One UI 6.0 (Android 14) on S10 — More difficult; camera and fingerprint often broken
One UI 6.1 / 7.0 on S10 — Extremely experimental; usually daily-driver unstable
Tools You'll Need
Table
Tool	Purpose
Odin3 / Heimdall	Flash Samsung firmware
SamFw Tool / Frija	Download official firmware from Samsung servers
APKTool / JADX	Decompile/modify One UI apps
imgtool / sdat2img	Extract/modify sparse images
Magisk	Root + systemless mods to fix compatibility
Play Integrity Fix	Pass Google Play integrity checks
The Harsh Reality
One UI ports are never as stable as official firmware.
Even the "best" S10 One UI 6.1 port will have:
Camera quality degradation (no Scene Optimizer, limited pro modes)
Broken or missing Samsung-exclusive features (Secure Folder, Knox, Pay)
Random reboots or sleep-of-death issues
Battery drain from unoptimized power management
Bluetooth/WiFi quirks
No OTA updates — manual flash every time
Daily driver? Only if you're tolerant of bugs. Most users keep a stable backup (stock One UI 2.5/3.1) and dual-boot or switch back when needed.
Better Alternatives for S10
If you want modern Android on the S10 with fewer headaches:
Table
Option	Pros	Cons
LineageOS 21/22 (Android 14/15)	Stable, clean, good performance	No Samsung features at all
crDroid	Feature-rich, customizable	No One UI look/feel
Evolution X	Pixel-like + customization	Same as above
Stock One UI 2.5/3.1 + Magisk modules	Most stable, all features work	Stuck on old Android version
Want to Try a Port?
Unlock bootloader (Exynos only — this wipes data and trips Knox permanently)
Flash TWRP/OrangeFox custom recovery
Find a reputable port on XDA with recent activity and user reports
Make a full backup of your current ROM via recovery
Flash the port (usually: wipe system/data/cache → flash ROM zip → flash Magisk → reboot)
Report bugs with logs to help the dev
